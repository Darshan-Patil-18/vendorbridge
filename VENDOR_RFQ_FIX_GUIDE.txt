VENDOR RFQ VISIBILITY FIX - IMPLEMENTATION GUIDE
================================================================================

PROBLEM FIXED:
--------------
Vendors could not see RFQs created by Procurement Officers because the system
was trying to match vendor IDs from the vendors table with user IDs from 
auth.users, which are different identifiers.

SOLUTION IMPLEMENTED:
--------------------
We implemented the simpler, more realistic approach where ALL registered 
vendors can see ALL available RFQs with status 'pending' or 'in_review'. 
This matches real-world procurement systems where any qualified vendor can 
submit quotes for available opportunities.

CHANGES MADE:
-------------

1. VENDOR MANAGEMENT PAGE (VendorManagement.jsx)
   ✅ Added "Vendor User ID (optional)" field
   ✅ Allows linking a vendor card to their login account
   ✅ Stores user_id when adding/editing vendors
   ✅ Field appears in the Add/Edit Vendor modal
   ✅ Helpful text explains the purpose

2. RFQ CREATION PAGE (RFQCreation.jsx)
   ✅ When officer creates RFQ and selects vendors:
      - Fetches user_id for each selected vendor
      - Stores vendor card IDs in selected_vendors array
      - Stores linked user IDs in selected_vendor_user_ids array
   ✅ Maintains existing vendor selection UI
   ✅ No UI changes - only data logic improved

3. QUOTATION SUBMISSION PAGE (QuotationSubmission.jsx)
   ✅ Changed from: Fetching only RFQs where vendor is selected
   ✅ Changed to: Fetching ALL RFQs with status 'pending' or 'in_review'
   ✅ Now ALL vendors can see ALL available RFQs
   ✅ Vendors can only quote once per RFQ (already quoted RFQs are disabled)
   ✅ More realistic procurement workflow

4. DATABASE MIGRATION (database_migration.sql)
   ✅ Created SQL script to add required columns
   ✅ Adds user_id to vendors table
   ✅ Adds selected_vendor_user_ids to rfqs table
   ✅ Adds performance indexes

-----------------

STEP 1: Run Database Migration

   4. Copy and paste the SQL into the editor
   5. Click "Run" to execute
   6. Verify the columns were added successfully

STEP 2: Deploy Code Changes
   1. Build the project:
      npm run build
   
   2. Deploy the dist folder to Netlify
      - Either push to your Git repo (auto-deploy)
      - Or manually upload the dist folder

STEP 3: Test the Fix
   1. Login as Procurement Officer
   2. Go to Vendor Management
   3. Edit a vendor and optionally add their User ID
   4. Create a new RFQ and select vendors
   
   5. Login as a Vendor
   6. Go to Quotation Submission page
   7. You should see ALL available RFQs
   8. Submit a quotation

OPTIONAL: Linking Vendors to Accounts
-------------------------------------
If you want to link vendor cards to user accounts:
1. Get the user's ID from Supabase Authentication > Users
2. Copy their UUID
3. Go to Vendor Management
4. Edit the vendor
5. Paste the UUID in "Vendor User ID (optional)"
6. Save

This linking is OPTIONAL. The system works perfectly without it.
All vendors see all RFQs regardless of linking.

TROUBLESHOOTING:
----------------
Issue: Vendors still don't see RFQs
Solution: 
  1. Check RFQ status is 'pending' or 'in_review'
  2. Verify database migration ran successfully
  3. Clear browser cache and refresh
  4. Check browser console for errors

Issue: Database migration fails
Solution:
  1. Make sure you have admin access to Supabase
  2. Run each ALTER TABLE statement separately
  3. Check if columns already exist before adding

Issue: Build fails
Solution:
  1. Delete node_modules folder
  2. Run: npm install
  3. Run: npm run build
